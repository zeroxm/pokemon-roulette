# Milestone v1.2: Pokédex Visual Enhancement

**Status:** ✅ SHIPPED 2026-04-08
**Phases:** 6–7
**Total Plans:** 4

## Overview

v1.2 added visual and interactive depth to the Pokédex. The milestone delivered shiny tracking persistence, a mobile fullscreen fix for the Pokédex modal, a pulsing gold glow animation on won Pokémon cells, and a rich Pokémon detail modal — the most significant UI feature since v1.0. Clicking any seen or won Pokémon now opens a dedicated modal with official-artwork, localized name, Pokédex number, a shiny toggle (when applicable), and alternate form selection.

## Milestone Summary

**Key Decisions:**

- Component-based modal (not template-based) — opened via `modalService.open(PokedexDetailModalComponent)`, keeps modal state (shiny toggle, selected form) inside the modal component
- `displayUrl` getter wraps `artworkUrl` with `hasError` fallback to `unknown.png` on 404
- `hasForms(PokemonItem)` avoided — use `getFormIds(id).length > 1` (only pokemonId available in detail modal)
- Alt-form `getPokemonById(formId)` avoided for IDs > 10000 (returns undefined) — use `PokemonForm.text` directly
- Mobile fix in `src/styles.css` global (not component-scoped) — ng-bootstrap renders modals in portal outside component scope

**Issues Resolved:**

- `markWon` was silently dropping `shiny: true` flag by object replacement — fixed with spread operator
- Test count arithmetic errors in plans caught by plan checker before execution (prevented false acceptance failures)
- Template/implementation inconsistency in artwork fallback (`[src]="artworkUrl"` vs `displayUrl`) caught and fixed in plans

**Issues Deferred:**

- Real-time flip animation when Pokédex modal is already open and Pokémon is caught
- Pokédex completion reward or notification
- Per-generation progress breakdown in National Dex
- Pokédex filtering/search
- Full Pokédex entries (types, stats, descriptions)

**Technical Debt Incurred:**

- `PokemonFormsService.getPokemonForms(pokemon: PokemonItem)` API requires full PokemonItem — a future refactor to accept `pokemonId` directly would simplify callers
- Official-artwork loaded per modal open (no pre-cache) — acceptable for v1.2 given lazy loading
