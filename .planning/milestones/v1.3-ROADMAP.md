# Milestone v1.3: Storage PC Regression Fix

**Status:** ✅ SHIPPED 2026-04-18
**Phases:** 12
**Total Plans:** 1

## Overview

A focused regression fix restoring the Storage PC drag-and-drop team editing feature that was broken in v1.1 Phase 8. When `getTeam()`/`getStored()` were made immutable (returning copies), the CDK DragDrop results in `StoragePcComponent` were mutating local copies that were never written back to the service. Added a `commitTeamAndStorage()` write-back API and wired it into `drop()`.

## Phases

### Phase 12: Storage PC Regression Fix

**Goal**: Dragging Pokémon between the trainer team and PC storage in the storage modal correctly updates the service's internal state, the trainer team display, and battle win odds — restoring behavior broken by the IMMUT-01 immutability fix in v1.1 Phase 8.
**Depends on**: Phase 11
**Plans**: 1 plan

Plans:

- [x] 12-01: Add `commitTeamAndStorage()` to TrainerService, fix `StoragePcComponent.drop()` write-back

**Details:**
- Root cause: `getTeam()` returns `[...this.trainerTeam]` (copy). CDK DragDrop mutated the local copies correctly but `updateTeam()` re-broadcast from the service's unchanged internal array — drag results were lost.
- Fix: new `commitTeamAndStorage(team, stored)` method writes arrays back into service internals before broadcasting via `trainerTeamObservable`
- 3 unit tests added for the new method (state write, observable emit, array-level immutability)

## Milestone Summary

**Key Decisions:**

- Decision: Add write-back API rather than reverting IMMUT-01 (Rationale: Immutable accessors are architecturally correct; the bug is a missing write path, not an immutability problem)
- Decision: Keep `updateTeam()` unchanged (Rationale: Other callers still need it; `commitTeamAndStorage()` is additive)
