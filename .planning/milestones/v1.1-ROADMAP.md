## Milestone Summary

**Milestone Stats:**
- Phases: 7–10 (4 phases)
- Plans: 12 total
- Files modified: 64
- LOC delta: +5,631 / -530
- Timeline: 2026-04-17 (single session)
- Tests: 177 → 217 (+40)

**Key Decisions:**
- `BaseBattleRouletteComponent` as plain abstract TypeScript class (no `@Component`) — Angular decorators not inheritable
- `usePotion` lambda pattern — keeps `ModalQueueService` out of base class; gym/E4 pass queue lambda, champion passes direct modal lambda
- `GENERATION_GAME_CONFIG` constant for all 9 generations — adding a new gen requires no code change in `GameStateService`
- `takeUntilDestroyed()` in constructor injection context — no explicit `DestroyRef` needed
- All battle component specs use `(component as any)` to access `protected` members from base class

**Issues Resolved:**
- Memory leaks in `RestartGameButtonComponent` and `SettingsButtonComponent` (never unsubscribed)
- Runtime crash potential in `BadgesService` on out-of-range round index
- O(n) `getPokemonById()` over 14k+ entries on every call → O(1) via Map
- 454 lines of verbatim duplication across 4 battle components
- Pre-existing copy-paste bug: elite-four i18n key used `gym` namespace

**Issues Deferred:**
- Shiny propagation TODOs in `pokedex.service.ts` — intentional, future PR
- `RouletteContainerComponent` full refactor — already decomposed enough; tracked
- `GameStateService` desync guard — theoretical, very low probability
- Language selector image-based flags — needs image assets

**Technical Debt Incurred:**
- `trainerItems` in base class captured once in `ngOnInit` (not reactive) — acceptable per current service contract
- Modal-service split (gym/E4 → `ModalQueueService`, champion/rival → `NgbModal`) is undocumented in code — acceptable, noted in review
