## Critical Gap: Double Type-Advantage Modal on Multi-Slot Leaders

### Affected encounters
- **Gym:** Gen 5 Round 0 (Cilan/Chili/Cress), Gen 7 Round 2 (Lana/Kiawe/Mallow), Gen 7 Round 4 (Sophocles/Acerola), Gen 8 Round 3 (Bea/Allister), Gen 8 Round 5 (Gordie/Melony)
- **E4:** Gen 8 Round 0 (Marnie/Hop/Bede), Gen 8 Round 2 (Bea/Allister)

### Root cause
In `getCurrentLeader()` / `getCurrentElite()`, the multi-slot path launches an async `translate.get()` subscription and returns. The outer `calcVictoryOdds()` + `queueTypeAdvantageModal()` in the `gameSubscription` handler then run immediately — at that point `currentLeader.types` still holds the **full multi-element array** (e.g. `['grass','fire','water']`), producing a synthetic matchup against all possible types. A first modal is queued.

When the async callback fires, `currentLeader` is replaced with the resolved single-leader object and `calcVictoryOdds()` + `queueTypeAdvantageModal()` run again — this time correctly with `[types[randomIndex]]`. A second modal is queued.

The user sees: `gymLeaderPresentation → typeAdvantage (wrong multi-type) → typeAdvantage (correct single-type)`

### Impact
- Wheel odds are correct after async resolves (second `calcVictoryOdds()` call overwrites the first)
- Modal content (strongCount/weakCount/label) in the first modal may be misleading
- Not a crash — the second modal has correct data
- Affects ~10% of leader slots (7/72 gym, 2/36 E4)

### Fix
Move the outer `calcVictoryOdds()` + `queueTypeAdvantageModal()` calls (in the `gameSubscription` handler) to **inside** the async callback for multi-slot leaders. Single-slot leaders (which don't enter the async block) retain the outer calls.
